# Monitoring — Setup Guide

Drift monitoring for the crop disease detection model using Evidently + Grafana.

---

## Architecture

```
API (FastAPI/Lambda)
        │  logs predictions (when MONITORING_DB_URL is set)
        ▼
PostgreSQL — predictions table
        │
        │  daily (Airflow monitoring_dag)
        ▼
src/monitoring/drift.py
        │  computes Evidently drift metrics
        ▼
PostgreSQL — drift_metrics table
        │
        ▼
Grafana dashboard — http://localhost:3000
```

---

## What is monitored

| Metric | Description | Threshold |
|---|---|---|
| `share_of_drifted_columns` | Fraction of features with drift | > 0.15 |
| `num_drifted_columns` | Count of drifted features | > 5 |
| `share_missing_values` | Missing values in current data | > 0.05 |

Features monitored: `predicted_class`, `confidence`, `entropy`

Reference data = test-set predictions from training (generated once, stored in S3).
Current data = recent API predictions logged to PostgreSQL (last 7 days).

---

## Prerequisites

- Docker and Docker Compose
- PostgreSQL monitoring stack running (see below)
- Reference data in S3 (generated by training — see Step 3)
- `.env` file with `MODEL_BUCKET` set

---

## Step 1 — Start the monitoring stack

```bash
docker compose -f monitoring/docker-compose.yaml up
```

Services started:
- **PostgreSQL** on port `5432` — metrics store
- **Adminer** on port `8080` — DB inspector (http://localhost:8080)
- **Grafana** on port `3000` — dashboard (http://localhost:3000, admin/admin)

Stop the stack:
```bash
docker compose -f monitoring/docker-compose.yaml down
```

---

## Step 2 — Enable prediction logging in the API

Set `MONITORING_DB_URL` when running the inference container to log each
prediction to PostgreSQL automatically:

```bash
docker run -p 8000:8000 \
    --entrypoint uvicorn \
    -v ~/.aws:/root/.aws:ro \
    -e RUN_ID=your_run_id \
    -e MLFLOW_TRACKING_URI=http://your-ec2:5000 \
    -e MONITORING_DB_URL=postgresql://POSTGRES_USER:POSTGRES_PASSWORD@localhost:5432/POSTGRES_DB \
    --network host \
    crop-disease-inference:latest \
    api.main:app --host 0.0.0.0 --port 8000
```

where you can find POSTGRES_USER, POSTGRES_PASSWORD and POSTGRES_DB in the docker-compose file.
You can send a predict request with the command:

```bash
curl -X POST http://localhost:8000/predict \
    -F "file=@data/processed/test/Tomato___healthy/000000.jpg"
```

Predictions are then written to the `predictions` table silently — the API
continues normally even if the DB is unreachable.

## Step 3 — Generate reference data (once after training)

Run Cell 10 in `notebooks/train_colab.ipynb` after training. This uploads
`monitoring/reference_predictions.parquet` to S3.

To verify it's there:
```bash
aws s3 ls s3://crop-disease-models-stg-478544568263/monitoring/
```

---

## Step 4 — Run drift monitoring manually

Quick test (no DB, uses synthetic data):
```bash
python src/monitoring/drift.py --skip-db
```

Full run against real DB (reads last 7 days of predictions):
```bash
MONITORING_DB_HOST=localhost   \
MONITORING_DB_PORT=5432   \
MONITORING_DB_NAME=monitoring \
MONITORING_DB_USER=postgres    \
MONITORING_DB_PASSWORD=example  \
MODEL_BUCKET=crop-disease-models-stg-478544568263 \
python src/monitoring/drift.py --window-days 7
```
where all the monitoring environment variables should match with what you set in Step 1 and 2
(localhost, port 5432, POSTGRES_USER, POSTGRES_PASSWORD and POSTGRES_DB)

Or via the Airflow scheduler (runs automatically every day at 6am):
```bash
docker compose -f airflow/docker-compose.yaml exec airflow-scheduler \
    airflow dags trigger crop_disease_monitoring
```

## Step 5 — View the Grafana dashboard

1. Open http://localhost:3000
2. Log in with `admin` / `admin`
3. Go to **Dashboards** → **Crop Disease Detection — Model Monitoring**

The dashboard has 8 panels:

| Panel | Query |
|---|---|
| Latest drift score | Most recent `share_of_drifted_columns` |
| Drifted columns count | Most recent `num_drifted_columns` |
| Missing values share | Most recent `share_missing_values` |
| Drift score over time | `drift_metrics` time series |
| Drifted columns over time | `drift_metrics` time series |
| Average confidence over time | Hourly avg from `predictions` |
| Class distribution (pie) | Last 7 days from `predictions` |
| Prediction volume | Hourly count from `predictions` |

Panels turn **yellow** at warning thresholds and **red** at critical thresholds.

## Database schema

```sql
-- populated by the API on each prediction
CREATE TABLE predictions (
    id               SERIAL PRIMARY KEY,
    timestamp        TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    predicted_class  VARCHAR(100),
    confidence       FLOAT,
    entropy          FLOAT,
    run_id           VARCHAR(64)
);

-- populated daily by src/monitoring/drift.py
CREATE TABLE drift_metrics (
    id                    SERIAL PRIMARY KEY,
    timestamp             TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    prediction_drift      FLOAT,
    num_drifted_columns   INTEGER,
    share_missing_values  FLOAT,
    window_size           INTEGER
);
```

Tables are created automatically on first run by `drift.py`.
To inspect via Adminer: http://localhost:8080
- System: PostgreSQL
- Server: db
- Username: postgres
- Password: example
- Database: monitoring

## Troubleshooting

| Error | Fix |
|---|---|
| Grafana shows "No data" | Run `drift.py` at least once to populate `drift_metrics` |
| `predictions` table empty | Set `MONITORING_DB_URL` env var when running the API |
| Reference data not found in S3 | Run Cell 10 in `train_colab.ipynb` after training |
| Port 5432 already in use | Stop the Airflow stack first (uses same port) or change monitoring port |
| DB connection refused | Check `MONITORING_DB_HOST` env var matches your setup |
